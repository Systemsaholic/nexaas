# Fleet Protocol

How each Nexaas client VPS reports its framework version and worker
health to the central Nexmatic ops dashboard.

This document defines the wire contract. The VPS side is implemented in
`packages/runtime/src/fleet/heartbeat.ts` and `packages/cli/src/init.ts`.
The central (Nexmatic) side lives in `/opt/nexmatic/` and must implement
the endpoints described here.

## Overview

```
┌─────────────────────┐                         ┌──────────────────────┐
│   Client VPS        │                         │   Nexmatic Ops       │
│   (Nexaas install)  │                         │   Console (central)  │
│                     │    1. one-time          │                      │
│ nexaas init --      │───  POST /register ────▶│  validate + issue    │
│   --fleet-endpoint  │◀── 200 {token} ─────────│    fleet token       │
│   --bootstrap-secret│                         │                      │
│                     │                         │                      │
│ worker starts       │    2. every 5 min       │                      │
│ fleet heartbeat ────┼─── POST /heartbeat ────▶│  upsert fleet row    │
│                     │    Bearer <token>       │  update dashboard    │
│                     │◀── 200 {ok:true} ───────│                      │
└─────────────────────┘                         └──────────────────────┘
```

## Environment variables (VPS side)

| var | purpose |
|-----|---------|
| `NEXAAS_FLEET_ENDPOINT` | base URL of the fleet API, e.g. `https://nexmatic.ca/api/fleet` |
| `NEXAAS_FLEET_TOKEN`    | workspace-scoped bearer token (issued by `POST /register`) |
| `NEXAAS_WORKSPACE`      | workspace id (already required for Nexaas runtime) |
| `NEXAAS_ROOT`           | framework install path (default `/opt/nexaas`) |

If `NEXAAS_FLEET_ENDPOINT` or `NEXAAS_FLEET_TOKEN` is unset, the
heartbeat loop maintains the local `framework_heartbeat` state row but
skips the remote push silently. Backwards compatible with pre-fleet
installs.

## Endpoint: `POST {endpoint}/register`

One-time registration, consumes a single-use bootstrap secret and
returns a permanent workspace-scoped bearer token.

### Request

- **Auth:** none (bootstrap_secret is the auth)
- **Content-Type:** `application/json`
- **Body:**

```json
{
  "workspace": "phoenix-voyages",
  "bootstrap_secret": "bs_2f9c8a7e3b1d4f6a...",
  "hostname": "phoenix.vps.ovh.net"
}
```

### Response (200)

```json
{
  "token": "flt_Ha7kXm...",
  "endpoint": "https://nexmatic.ca/api/fleet"
}
```

### Response (4xx)

| status | when |
|-------|------|
| 400 | missing/malformed fields |
| 401 | unknown or already-consumed bootstrap_secret |
| 409 | workspace already registered (caller must rotate the stored token out of band) |

Body on error:

```json
{ "error": "bootstrap_secret already consumed" }
```

### Server-side behaviour (Nexmatic)

1. Look up `fleet_bootstrap_secrets.secret = $secret` with `consumed_at IS NULL`
2. Verify `workspace` matches the secret's scope
3. Generate a high-entropy token (e.g. `flt_` + 32 bytes base64url)
4. Insert into `fleet_tokens (workspace, token_hash, issued_at, hostname)` — store only the SHA-256 hash of the token
5. Mark the bootstrap secret consumed
6. Return the plaintext token once — never retrievable again

Bootstrap secrets are generated by an operator in the ops-console when
creating a workspace, shown once, and pasted into the VPS during
`nexaas init`.

## Endpoint: `POST {endpoint}/heartbeat`

Liveness + version signal. Called every 5 min from the worker, plus on
startup and after `nexaas upgrade`.

### Request

- **Auth:** `Authorization: Bearer <NEXAAS_FLEET_TOKEN>`
- **Content-Type:** `application/json`
- **Body:**

```json
{
  "workspace": "phoenix-voyages",
  "version": "0.2.0",
  "commit_sha": "44bf213",
  "branch": "main",
  "hostname": "phoenix.vps.ovh.net",
  "started_at": "2026-04-19T10:15:22.431Z",
  "now": "2026-04-19T10:20:22.431Z",
  "worker_status": "running"
}
```

Field notes:
- `started_at` — when the worker process booted. Useful for uptime calc.
- `now` — client clock at send time. Compare to server-side `received_at`
  to detect clock skew.
- `worker_status` — always `"running"` in v1. Future: `"draining"`,
  `"degraded"`, with subfields for reason.

### Response (200)

```json
{ "ok": true }
```

### Response (4xx)

| status | when |
|-------|------|
| 401 | missing/invalid token |
| 403 | token exists but doesn't match `workspace` in body |
| 404 | workspace not provisioned on the fleet |

VPS side treats any non-2xx as a push failure: logs the status in the
local `framework_heartbeat` row and a WAL `framework_heartbeat_failed`
entry, but **never crashes the worker**.

### Server-side behaviour (Nexmatic)

1. Extract bearer token; look up `fleet_tokens` by SHA-256 hash
2. Verify `tokens.workspace === body.workspace`
3. Upsert into `fleet_heartbeats (workspace, version, commit_sha, branch, hostname, started_at, received_at, worker_status)`
4. Return `200 {ok:true}`

For the dashboard `/fleet` page, query the latest heartbeat per
workspace. Anything with `now() - received_at > 15 min` is "stale"
(yellow badge); `> 1 hr` is "dead" (red).

## Suggested Nexmatic schema

```sql
CREATE TABLE nexmatic.fleet_workspaces (
  workspace        text PRIMARY KEY,
  display_name     text,
  created_at       timestamptz NOT NULL DEFAULT now(),
  provisioned_by   text
);

CREATE TABLE nexmatic.fleet_bootstrap_secrets (
  secret           text PRIMARY KEY,
  workspace        text NOT NULL REFERENCES nexmatic.fleet_workspaces(workspace),
  created_at       timestamptz NOT NULL DEFAULT now(),
  consumed_at      timestamptz,
  expires_at       timestamptz NOT NULL   -- recommend 24h
);

CREATE TABLE nexmatic.fleet_tokens (
  token_hash       text PRIMARY KEY,       -- sha256(plaintext)
  workspace        text NOT NULL REFERENCES nexmatic.fleet_workspaces(workspace),
  hostname         text,
  issued_at        timestamptz NOT NULL DEFAULT now(),
  last_used_at     timestamptz,
  revoked_at       timestamptz
);

CREATE TABLE nexmatic.fleet_heartbeats (
  workspace        text PRIMARY KEY REFERENCES nexmatic.fleet_workspaces(workspace),
  version          text,
  commit_sha       text,
  branch           text,
  hostname         text,
  started_at       timestamptz,
  received_at      timestamptz NOT NULL DEFAULT now(),
  worker_status    text
);

-- Append-only audit if you want fleet history
CREATE TABLE nexmatic.fleet_heartbeat_log (
  id               bigserial PRIMARY KEY,
  workspace        text NOT NULL,
  version          text,
  commit_sha       text,
  received_at      timestamptz NOT NULL DEFAULT now()
);
```

## Rollout order

1. Deploy this framework version (`v0.2.0+`) to Phoenix — heartbeat module
   exists but is a no-op because fleet env vars aren't set yet.
2. Implement `/api/fleet/register` and `/api/fleet/heartbeat` on the
   Nexmatic ops-console.
3. Generate a bootstrap secret for Phoenix in the ops-console.
4. On Phoenix: add `NEXAAS_FLEET_ENDPOINT` + `NEXAAS_FLEET_TOKEN` to
   `.env` (or re-run `nexaas init --fleet-endpoint … --bootstrap-secret …`),
   restart the worker. First heartbeat lands within ~5s.
5. Verify on dashboard, then do the same on subsequent client VPSes.

## Security notes

- Tokens are stored hashed on the central side — DB leak doesn't leak
  plaintext tokens. VPS side stores plaintext in `.env` (chmod 600).
- Bootstrap secrets are single-use and time-bounded. A leaked secret
  that hasn't been consumed can be revoked in the ops-console.
- Transport must be HTTPS. The `fetch` call in the heartbeat module will
  fail TLS validation against self-signed certs — fine for v1, revisit
  if there's demand for on-prem deployments.
- Heartbeat push is fire-and-forget from the worker's perspective.
  Token invalidation on the central side does not affect the worker's
  ability to process jobs; it just hides the VPS from the fleet page.
